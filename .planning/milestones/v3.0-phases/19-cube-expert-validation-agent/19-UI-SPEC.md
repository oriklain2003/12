## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 13px | 400 | 1.5 |
| Label / UI text (issue row message) | 12px | 400 | 1.4 |
| Label / caption (cube name, param name badges) | 11px | 400 | 1.3 |
| Panel section heading | 11px | 600 | 1.3 |

Notes:
- 13px is the dominant UI body size in this project (highest frequency in CSS audit: 16 occurrences of `font-size: 13px`).
- 12px is the data-density size used in tables and status labels (18 occurrences).
- 11px is used for uppercase section labels and meta-information (22 occurrences — most common).
- No display or heading size (24px+) is needed in the issues panel or validation flow for this phase.
- Letter spacing: `-0.01em` for body text (matches `body` rule in `index.css`); `-0.025em` for any heading text.
- Font family: inherit from `body` — DM Sans. Exception: cube name and param name tokens in issue rows use `font-family: var(--font-mono, 'JetBrains Mono', 'Fira Code', monospace)` to distinguish identifiers from prose (matches `results-table` monospace pattern). The monospace font-family change provides sufficient visual distinction — no intermediate weight is needed.
- Two weights only: 400 (body, message text, label/UI text, monospace identifiers) and 600 (panel section heading).

Source: font-size frequency count across all frontend CSS files; weight values from `font-weight: 600` occurrences.

## Component Specification: IssuesPanel

This is the primary new UI component for Phase 19 (VALID-01, VALID-02, VALID-03, per CONTEXT.md D-07).

### Layout

```
┌─ Editor page (flex column) ────────────────────────────────────────┐
│  Toolbar (52px, fixed)                                              │
│  ┌─ Editor body (flex row, flex: 1) ──────────────────────────────┐│
│  │  Sidebar (fixed width)  │  FlowCanvas (flex: 1)                ││
│  └─────────────────────────────────────────────────────────────────┘│
│  IssuesPanel (collapsible, below canvas)                            │
└─────────────────────────────────────────────────────────────────────┘
```

The IssuesPanel sits below the canvas in document flow — not as an overlay or modal. It pushes the canvas upward when expanded (resizes the flex column). When collapsed, it shows only a 32px header bar.

### Panel States

| State | Description |
|-------|-------------|
| Hidden | Panel not rendered; zero issues and validation has not yet run |
| Collapsed | 32px header bar visible; shows issue count summary; canvas takes full remaining height |
| Expanded | Header + issue list visible; default expanded height 180px; canvas shrinks proportionally |
| Loading | Panel shows header with "Validating..." text and a subtle pulse animation during the `POST /api/agent/validate` network call (expected <100ms — the pulse is a safety net for slow connections only) |

### Header Row (32px height)

```
[ v ] Issues  [ 2 errors · 1 warning ]          [ ^ collapse ]
```

- Toggle chevron: `▼` expanded, `▶` collapsed; 11px, `--color-text-muted`
- Section heading: "Issues" in 11px, weight 600, uppercase, letter-spacing 0.06em, `--color-text-muted` — matches `.toolbar__shortcuts-title` pattern
- Count pills: inline `<span>` elements; errors use `--color-error` text, warnings use `--color-warning` text; separated by `·` in `--color-text-muted`
- Collapse toggle: right-aligned; 28px touch target; `▲` / `▼` chevron; same style as panel chevron

### Issue Row

Each issue is one row with:

```
[ icon ] [ cube-name ] › [ param-name ]   message text
```

- Left border: 2px solid `--color-error` for errors, 2px solid `--color-warning` for warnings
- Background: transparent; hover state `--color-surface-hover` (matches node hover pattern)
- Icon: `●` bullet 8px, colored per severity (`--color-error` or `--color-warning`)
- Cube name: 11px monospace, weight 400, `--color-text-dim`
- Separator `›`: 11px, `--color-text-muted`
- Param name: 11px monospace, weight 400, `--color-text-dim` (omitted when `param_name` is null — cycle errors, unknown cube errors)
- Message: 12px DM Sans, weight 400, `--color-text-secondary`; right of the identifier block, same line on wide viewports
- Padding: 8px top/bottom, 12px left/right
- Cursor: `pointer` — entire row is clickable; clicking highlights the relevant canvas node

### Click-to-Highlight Behavior

- Clicking an issue row dispatches a `highlightNode(nodeId)` action to the Zustand `flowStore`
- The highlighted node receives `--shadow-node-selected` box-shadow (existing token: `0 0 0 2px var(--color-accent), 0 8px 40px rgba(var(--accent-rgb), 0.2)`)
- React Flow `fitView` is called with the targeted node to ensure it is visible
- Highlight clears when the user clicks elsewhere on the canvas

### Validation States in Toolbar

Per CONTEXT.md D-08: clean validation is silent — no toast, no dialog. The only toolbar change is:

- While validating: Run button label changes to "Validating..." and is disabled (opacity 0.4, `cursor: not-allowed` — matches `.toolbar__btn:disabled`)
- On error block: Run button re-enables; issues panel opens automatically; no other toolbar change
- On clean pass: execution starts immediately; existing progress bar appears as normal

## Interaction Contracts

### Run Button Click Flow

1. User clicks "Run" in Toolbar.
2. Frontend calls `POST /api/agent/validate` with current workflow graph.
3. Run button becomes "Validating..." + disabled.
4. On response:
   - If `issues` array contains any `severity: "error"` items: execution is blocked, IssuesPanel opens expanded, Run button re-enables. No toast.
   - If `issues` array contains only `severity: "warning"` items: execution proceeds, IssuesPanel opens expanded (warnings visible), existing SSE stream starts.
   - If `issues` array is empty: execution proceeds immediately, IssuesPanel remains hidden.

### IssuesPanel Collapsibility

- Collapsed by default when first shown (user sees header bar indicating issue count; they must expand to read messages).
- Exception: when execution is blocked (error path), panel opens expanded automatically — the user must see why execution was blocked.
- Collapse state is local component state (not persisted to Zustand or server).

### Node Highlighting

- Clicking an issue row: calls `fitView({ nodes: [{ id: nodeId }], padding: 0.3 })` on the React Flow instance + applies selected class to the node.
- Only one node highlighted at a time.
- Highlight resets on canvas click (outside any node).
