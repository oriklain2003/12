  <action>
Create `engine/executor.py` with class `WorkflowExecutor`:

**Constructor:**
- Accepts `graph: WorkflowGraph` and stores it
- Imports `registry` from `app.engine.registry`
- Imports `settings` from `app.config` for `result_row_limit`

**`async execute(self, db_connection=None) -> dict`:**
Main execution method. Returns dict keyed by node_id with per-node results.

1. **Build adjacency graph:** Parse `graph.edges` to build:
   - `dependencies: dict[str, set[str]]` — for each node, which nodes must run first (source nodes of incoming edges)
   - `edge_map: dict[str, list[edge]]` — edges grouped by target node, for resolving which output param feeds which input param

2. **Topological sort with cycle detection (Kahn's algorithm):**
   - Count in-degrees for each node
   - Start with nodes that have in-degree 0
   - Process queue: for each node, decrement in-degrees of dependents
   - If processed count != total nodes, raise HTTPException(400, "Workflow graph contains a cycle")
   - Result: ordered list of node_ids to execute

3. **Execute nodes in topological order:**
   For each node_id in order:
   - Look up the cube via `registry.get(node.data.cube_id)`
   - **Resolve inputs:**
     a. Start with manual param values from `node.data.params` (user-entered values)
     b. For each edge targeting this node: get the source node's output for `edge.sourceHandle`, set it as the input for `edge.targetHandle` — this OVERRIDES any manual value (BACK-12)
     c. Special case: if `sourceHandle == "__full_result__"`, the value is the full outputs dict of the source node bundled as one JSON object (BACK-10)
     d. Type compatibility: log a warning if source output type != target input type (don't block execution — type mismatches are allowed per spec, just warned)
     e. Full Result validation: if sourceHandle is `__full_result__`, check that the target param has `accepts_full_result=True` in its ParamDefinition. If not, log warning but still allow (connections are lenient).
   - **Check if upstream failed:** If any dependency node has status "error" or "skipped", mark this node as `"status": "skipped", "reason": "Upstream node {failed_id} failed"` and continue to next node
   - **Execute cube:** Call `await cube.execute(**resolved_inputs)`, passing `db_connection` if the cube needs it
   - **Row limiting (BACK-11):** For each output value that is a list, if `len(value) > settings.result_row_limit`, truncate to first `result_row_limit` items and set `truncated = True`
   - **Store results:** `results[node_id] = {"status": "done", "outputs": outputs, "truncated": truncated}`
   - **On exception:** Catch any exception, store `results[node_id] = {"status": "error", "error": str(e)}`. Do NOT re-raise — continue executing other nodes (BACK-09 failure behavior: independent branches continue)

4. Return the results dict.

**Important design notes:**
- The executor does NOT access the database for workflow loading — it receives a `WorkflowGraph` object. The router handles loading from DB.
- The `db_connection` parameter is passed through to cubes that need database access (like all_flights). Cubes that don't need it ignore it.
- Use `BaseCube.execute(**inputs)` — the cube's execute method signature uses `**inputs` kwargs.
- For cubes that need database access, pass connection as a special kwarg (e.g., `_db_connection`). The BaseCube.execute signature uses `**inputs` so this gets absorbed. Individual cubes extract it from kwargs if they need it.
  </action>

2. Build the SQL query dynamically:
   - Base: `SELECT * FROM research.flight_metadata WHERE 1=1`
   - Time filter: If `time_start` provided, use absolute range. Otherwise use `time_range_hours` to compute epoch threshold: `WHERE created_at > extract(epoch from now() - interval '{hours} hours')` (check actual column name — it may be a bigint epoch column like `timestamp` or `created_at`)
   - Flight IDs: `WHERE flight_id = ANY(:flight_ids)` (pass as array parameter)
   - Callsign: `WHERE callsign ILIKE :callsign` (wrap with % for partial match)
   - Altitude: Join with `research.normal_tracks` to filter flights where ALL track points are between altitude_min and altitude_max. Use a subquery: `flight_id IN (SELECT DISTINCT flight_id FROM research.normal_tracks WHERE flight_id = ANY(candidate_ids) GROUP BY flight_id HAVING MIN(altitude) >= :alt_min AND MAX(altitude) <= :alt_max)`
   - Polygon: Join with `research.normal_tracks` to find flights with ANY position inside the polygon. Use PostgreSQL point-in-polygon: construct a polygon check using latitude/longitude columns. If PostGIS is available, use `ST_Contains(ST_MakePolygon(ST_MakeLine(array[ST_MakePoint(lon,lat)...])), ST_MakePoint(track.longitude, track.latitude))`. If PostGIS is NOT available, implement a ray-casting point-in-polygon check in Python on the returned track data, or use a simpler bounding-box approach with manual refinement.

3. Execute query using the raw connection: `result = await connection.execute(text(query), params)`
4. Convert rows to list of dicts
5. Extract flight_ids list from results
6. Return `{"flights": flights_list, "flight_ids": flight_ids_list}`

**IMPORTANT considerations:**
- The database is read-only. All queries are SELECT only.
- Use `sqlalchemy.text()` for raw SQL with bound parameters (no f-string SQL injection)
- Column names in research.flight_metadata need to be discovered. The executor may need to handle this gracefully if columns differ from expectations. Add a try/except around the query with a descriptive error message.
- The altitude and polygon filters require joining with research.normal_tracks which is a separate table with position data per flight. These are expensive queries — apply flight_metadata filters first to narrow candidates before joining normal_tracks.
- If the cube encounters a database error, raise an exception with a clear message — the executor's failure handling will catch it and mark the node as "error".
