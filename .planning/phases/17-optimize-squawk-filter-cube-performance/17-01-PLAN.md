**1. SQL Pushdown (per user decision):**
- FR provider: Replace `AND squawk IS NOT NULL` with `AND squawk = ANY(:codes)` in the SQL query. Add `"codes": list(target_codes)` to the params dict. This works for both custom mode (target_codes = user-specified codes) and emergency mode (target_codes = EMERGENCY_CODES_FR = {"7500", "7600", "7700"}) since target_codes is already set correctly in Step 2.
- Alison custom mode: Replace `AND squawk IS NOT NULL` with `AND squawk = ANY(:codes)` in the custom mode SQL query. Add `"codes": list(target_codes)` to the params dict.
- Alison emergency mode: Leave unchanged (already queries by `emergency IS NOT NULL` pattern, no squawk filtering needed).
- After SQL pushdown, replace the Python `any(r["squawk"] in target_codes for r in position_rows if r["squawk"])` matching check with `matched = len(position_rows) > 0` for all paths where SQL now guarantees only matching rows return (FR both modes, Alison custom mode). Keep the existing `any(...)` check only for Alison emergency mode where SQL does not filter by squawk code.

**2. Set-based accumulation (per user decision):**
- Change `codes_seen`, `matched_codes`, `emergency_values` from lists to sets during accumulation (`set()` instead of `[]`).
- Remove the `if code not in codes_seen` guard — just use `codes_seen_set.add(code)`.
- At output time, convert to sorted lists: `sorted(codes_seen_set)`, `sorted(matched_codes_set)`, `sorted(emergency_values_set)`.

**3. Loop hoisting (per user decision):**
- Before the per-row loop, compute: `is_emergency = (mode == "emergency")` and `is_alison = (provider == "alison")`.
- Replace in-loop `mode == "emergency"` with `is_emergency` and `provider == "alison"` with `is_alison`.

Do NOT modify any other behavior. Do NOT add new features. Do NOT change the Alison emergency SQL path.
